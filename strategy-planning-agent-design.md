# 사업/전략 기획 자동화 에이전트 — 통합 설계서

> 본 문서는 Claude Code에서 구체적인 구현 시 참조하는 계획서입니다.
> 상세 파일 내용(CLAUDE.md, AGENT.md, SKILL.md)은 구현 단계에서 작성합니다.

---

## 1. 작업 컨텍스트 문서

### 1.1 배경 및 목적

사업/전략 기획 과정에서 자료 수집·리서치, 문서 초안 작성, 아이디어 정리, 검토·피드백 반영의 네 단계가 반복적·시간 집약적으로 발생한다. 본 에이전트 시스템은 "키워드 한 줄" 수준의 최소 입력으로 시작해 외부 공유 가능한 완성도의 기획 문서를 자동 생성하는 것을 목적으로 한다. **본인이 직접 사용하는 개인 생산성 도구**이며, 사람은 세 번의 핵심 시점(인터뷰 결과 확인, 리서치 결과 확인, 초안 검토)에만 개입하여 품질을 통제한다.

**목표 품질 기준:** 외부 공유 가능한 수준 (80~90점). 에이전트가 생성한 초안에서 직접 수정해서 쓸 수 있는 수준.

### 1.2 범위

| 항목 | 포함 | 제외 |
|------|------|------|
| 입력 처리 | 키워드·테마 한 줄 파싱 및 컨텍스트 확보 인터뷰 | 음성 입력, 이미지 처리 |
| 리서치 | 웹 검색 기반 시장·경쟁사·외부환경 조사 (병렬 3트랙 동시 실행) | 유료 DB, 내부 데이터 연동 (v2 확장) |
| 전략 도출 | SWOT, 포지셔닝, 전략 옵션 생성 | 재무 모델링, 정량 시뮬레이션 |
| 문서 생성 | 기획서 (v1 주력), 사업계획서·시장분석보고서·제안서 (기본 템플릿 제공) | 슬라이드 자동 생성 (v2 확장) |
| 사람 개입 | 인터뷰 결과 확인, 리서치 완료 후, 초안 완료 후 각 1회 | 실시간 스트리밍 편집 |
| 출력 포맷 | Markdown 전용 | docx 변환 (v2 확장) |
| 세션 복원 | /output 파일 존재 체크로 중단 시점 자동 탐지 후 재개 | — |

### 1.3 입출력 정의

**입력**
- 필수: 키워드/테마 한 줄 (예: `"국내 시니어 헬스케어 시장 진출 전략"`)
- 선택 (인터뷰에서 확보): 산출물 유형, 대상 독자, 강조 관점, 지역/기간 범위, **분야 변화 속도(최신성 기준 동적 설정용)**

**출력** (사용자가 선택한 유형, v1 기획서 집중)

| 유형 | 주요 섹션 | v1 상태 |
|------|----------|---------|
| 기획서 | 목적·배경, 시장 현황, 전략 방향, 실행 계획 | **주력 — 완성도 우선** |
| 사업계획서 | 사업 개요, 시장·경쟁 분석, 수익 모델, 로드맵 | 기본 템플릿 제공 |
| 시장/경쟁사 분석 보고서 | 시장 규모·트렌드, 경쟁사 매핑, 기회·위협 | 기본 템플릿 제공 |
| 제안서 | 문제 정의, 솔루션, 기대 효과, 다음 단계 | 기본 템플릿 제공 |

**중간 산출물** (에이전트 내부용, `/output/` 저장)

| 파일 | 내용 |
|------|------|
| `step1_context.json` | 인터뷰로 확보한 컨텍스트 구조체 |
| `step2_research.json` | 리서치 결과 구조화 데이터 |
| `step3_strategy.json` | 분석 결과 및 전략 옵션 |
| `step4_draft.md` | 문서 초안 (하단에 `<!-- FEEDBACK_LOG -->` 섹션 포함) |

### 1.4 주요 제약조건

- **데이터 소스:** Claude 내장 WebSearch (v1) / Tavily MCP는 v2에서 확장
- **문서 포맷:** Markdown 전용 (docx 변환은 v2)
- **최대 리서치 쿼리:** 단계당 5회 이내 — 쿼리를 정보량 기준으로 설계하여 5회 안에 최대 커버리지 확보
- **최신성 기준:** 인터뷰에서 분야 변화 속도를 파악하여 동적 설정 (예: AI 시장 = 6개월, 전통 제조업 = 3년). `step1_context.json`의 `freshness_months` 필드로 저장
- **병렬 처리:** researcher 에이전트 내에서 3개 트랙을 병렬 Task로 동시 실행 (속도 필수)
- **사람 개입 없이 자동 완료되는 단계:** 리서치, 전략 도출
- **사람 개입이 반드시 필요한 단계:** 인터뷰 결과 확인, 리서치 결과 확인, 초안 검토

### 1.5 용어 정의

| 용어 | 정의 |
|------|------|
| **컨텍스트** | 에이전트가 작업 수행에 필요한 배경 정보 (산출물 유형, 독자, 제약 등) |
| **에스컬레이션** | 에이전트가 판단하기 어려운 경우 사용자에게 추가 정보를 직접 요청하는 행위 |
| **Two-pass 검증** | 에이전트가 생성 역할로 한 번, 비판 역할로 한 번 — 두 패스로 자기 검토하는 과정 |
| **중간 확인** | 사람이 개입하여 방향을 확인하고 수정 지시를 내리는 체크포인트 |
| **FEEDBACK_LOG** | step4_draft.md 하단에 HTML 코멘트로 삽입되는 피드백 이력. writer 재실행 시 이 이력 전체를 읽고 반영 |

## 2. 워크플로우 정의

### 2.1 전체 흐름도

```
[START]
  → /output 폴더 파일 존재 체크 → 이미 완료된 단계 자동 탐지 → 해당 시점부터 재개
    │ (파일 없으면 처음부터 시작)
    ▼
[Phase 1: 인터뷰]  ← LLM 판단
  1) 키워드로 미니 사전 스캔 (2~3 쿼리) — 이미 공개 정보로 알 수 있는 내용 파악
  2) 모르는 것 중 가장 중요한 3가지만 질문 (산출물 유형, 독자, 분야 변화 속도 등)
  → step1_context.json 저장 (freshness_months 포함)
    │
    ▼
[✅ 인터뷰 결과 확인]  ← 사람 개입 #1
  "이렇게 이해했습니다:
    - 산출물: 기획서
    - 독자: 내부 경영진
    - 최신성 기준: 1년 이내
    - 기본값으로 설정한 항목: ..."
  → 사용자 수정/승인 → step1_context.json 업데이트
    │
    ▼
[Phase 2: 리서치]  ← 병렬 웹 검색 (3트랙 동시)
  트랙 A: 시장 규모·트렌드
  트랙 B: 경쟁사 현황·포지셔닝
  트랙 C: 외부환경 (규제·기술·소비자)
  → step2_research.json 저장
    │
    ▼
[🙋 중간 확인 1: 리서치 검토]  ← 사람 개입 #2
  형식: 주요 발견 가설 3개 + 각 근거
  (데이터 충돌 있으면 명시: "사용자 의도 vs 시장 현실이 다릅니다 — 방향을 선택해주세요")
  ├── 수정 있음 → 해당 트랙 재조사 (최대 1회) → 재요약 제시
  └── 확인 → 다음 단계
    │
    ▼
[Phase 3: 분석·전략 도출]  ← LLM Two-pass
  1차: SWOT 분석 → 포지셔닝 맵 → 전략 옵션 3개 도출
       (다양성 제약: 3개는 리스크·투자 규모·진입 타이밍 기준으로 서로 달라야 함)
  2차: 동일 에이전트가 비판자 역할로 재검토
       (논리 일관성, 근거 충분성, 다양성 확인)
  → step3_strategy.json 저장
    │
    ▼
[Phase 4: 문서 초안 생성]  ← LLM 생성
  입력: step1_context.json + step2_research.json + step3_strategy.json
  산출물 유형에 맞는 구조로 초안 작성 (원본 수치·출처 인용 가능)
  → Two-pass 자기 검증 (섹션 완성도, 톤 일관성, 독자 적합성)
  → step4_draft.md 저장 (하단에 빈 <!-- FEEDBACK_LOG --> 섹션 포함)
    │
    ▼
[🙋 중간 확인 2: 초안 검토]  ← 사람 개입 #3
  초안 전체 제시 → 사용자 피드백 수집
  ├── 피드백 있음 → step4_draft.md의 FEEDBACK_LOG에 이력 추가
  │                → writer 재실행 시 FEEDBACK_LOG 전체 읽고 모든 피드백 반영
  │                → 수정 초안 재제시 (반복 횟수 제한 없음, 사용자 승인까지)
  └── 승인 → 최종 문서 생성
    │
    ▼
[Phase 5: 최종 산출물 저장]
  /output/final_<type>_<date>.md (Markdown 전용, docx는 v2)
    │
    ▼
[END]
```

### 2.2 LLM 판단 영역 vs 코드 처리 영역

| 단계 | LLM 판단 영역 | 코드/스크립트 처리 영역 |
|------|--------------|----------------------|
| 인터뷰 | 미니 스캔 기반 질문 선정, 충분한 컨텍스트 확보 여부 판단, 기본값 설정 및 명시 | context.json 스키마 검증 |
| 리서치 | 검색 쿼리 설계(정보량 최적화), 결과 신뢰도·관련성 평가, 중복 제거, 충돌 감지 | 웹 검색 API 호출, 결과 파싱, 파일 저장 |
| 분석·전략 | SWOT 도출, 포지셔닝 판단, 전략 옵션 우선순위 및 다양성, Two-pass 비판 검토 | JSON 구조화, 파일 저장 |
| 문서 생성 | 서술 품질, 톤·어조, 섹션 완성도, FEEDBACK_LOG 반영, Two-pass 자기 검증 | 마크다운 포맷팅 |
| 자기 검증 | 논리 일관성, 누락 섹션, 데이터 근거 충분성, 전략 옵션 다양성 평가 | 스키마 검증, 필수 섹션 존재 확인 |
| 세션 복원 | — | /output 파일 존재 체크, 완료 단계 자동 탐지 |

### 2.3 각 단계 성공 기준 및 검증

#### Phase 1: 인터뷰

| 항목 | 내용 |
|------|------|
| **성공 기준** | 산출물 유형, 대상 독자, 핵심 관점/제약, freshness_months 4개 항목이 모두 확정된 상태 |
| **인터뷰 선정 로직** | 인터뷰 전 키워드로 미니 스캔(2~3 쿼리) → 공개 정보로 알 수 있는 항목 제외 → 모르는 것 중 가장 중요한 최대 3가지만 질문 |
| **인터뷰 결과 확인** | 인터뷰 종료 후 반드시 "이렇게 이해했습니다 — 맞나요?" 요약 메시지 출력. 기본값으로 설정한 항목을 명시적으로 보여줌 |
| **검증 방법** | 스키마 검증 (step1_context.json의 필수 필드 존재 여부) |
| **실패 시 처리** | 누락 항목에 대해 추가 질문 자동 재시도 (최대 3회) → 여전히 불명확하면 LLM이 합리적 기본값 적용 후 "이렇게 이해했습니다" 확인 단계에서 명시 |

#### Phase 2: 리서치

| 항목 | 내용 |
|------|------|
| **성공 기준** | 3개 트랙 각각 최소 3건 이상의 신뢰도 있는 출처에서 핵심 데이터 확보 |
| **병렬 실행** | 3개 트랙을 동시에 병렬 Task로 실행 (순차 금지) |
| **쿼리 설계** | 트랙당 최대 5회 쿼리 — 정보량·커버리지 기준으로 쿼리를 설계하여 5회 안에 최대 확보 |
| **최신성 기준** | step1_context.json의 freshness_months 값 적용 (고정 2년 아님) |
| **검증 방법** | LLM Two-pass: 생성 후 비판 역할로 재검토 (출처 다양성, 데이터 최신성, 핵심 질문 커버리지) |
| **충돌 처리** | 사용자 의도와 리서치 데이터가 충돌하면 중간 확인 1에서 명시 후 사용자가 방향 선택 |
| **실패 시 처리** | 특정 트랙 데이터 부족 시 다른 키워드로 재검색 (최대 2회) → 이후에도 부족하면 "데이터 제한" 명시 후 중간 확인 1로 진행 |

#### 중간 확인 1 (리서치 검토)

| 항목 | 내용 |
|------|------|
| **성공 기준** | 사용자가 리서치 방향을 승인하거나 구체적 수정 지시를 제공 |
| **제시 형식** | 주요 발견 가설 3개 + 각 근거 (충돌 있으면 별도 섹션으로 명시 + 방향 선택 요청) |
| **검증 방법** | 사람 검토 |
| **실패 시 처리** | 수정 지시 수령 시 해당 트랙 재조사 (최대 1회) → 재조사 결과를 다시 요약 제시 |

#### Phase 3: 분석·전략 도출

| 항목 | 내용 |
|------|------|
| **성공 기준** | SWOT 4분면 완성, 전략 옵션 3개 이상 도출, 각 옵션에 데이터 기반 근거 1개 이상 명시 |
| **전략 다양성 제약** | 3개 옵션은 리스크·투자 규모·진입 타이밍 기준으로 서로 달라야 함 (유사 옵션 3개 금지) |
| **검증 방법** | Two-pass: 1차 생성 → 2차 동일 에이전트가 비판자 역할로 재검토 (근거-주장 일관성, SWOT 항목 공백, 다양성) |
| **에스컬레이션** | 검증 2회 실패 또는 판단 불가 상황 → 사용자에게 추가 정보 직접 요청 |

#### Phase 4: 문서 초안 생성

| 항목 | 내용 |
|------|------|
| **성공 기준** | 선택한 산출물 유형의 필수 섹션이 모두 존재하고, 각 섹션이 200자 이상 |
| **writer 입력** | step1_context.json + step2_research.json (원본 수치·출처 인용용) + step3_strategy.json |
| **검증 방법** | 규칙 기반 (필수 섹션 체크) + LLM Two-pass (톤 일관성, 독자 적합성) |
| **피드백 루프 메커니즘** | 피드백 수렴 시 step4_draft.md 하단 `<!-- FEEDBACK_LOG -->` 섹션에 이력 추가. writer 재실행 시 FEEDBACK_LOG 전체를 읽고 모든 피드백을 반영한 뒤 초안 수정 |
| **실패 시 처리** | 누락 섹션 자동 보완 후 재검증 (최대 2회) |

#### 중간 확인 2 (초안 검토)

| 항목 | 내용 |
|------|------|
| **성공 기준** | 사용자가 초안을 승인하거나 구체적 수정 사항 제시 |
| **검증 방법** | 사람 검토 |
| **피드백 반영** | FEEDBACK_LOG 누적 방식으로 이전 피드백 이력 보존. writer 재실행 시 전체 이력 읽고 반영 (A 피드백 반영 시 B 섹션 회귀 방지) |
| **실패 시 처리** | 피드백 기반 해당 섹션 수정 → 전체 재제시 (반복 횟수 제한 없음, 사용자 승인까지) |

#### Phase 5: 최종 산출물

| 항목 | 내용 |
|------|------|
| **성공 기준** | 최종 파일이 `/output/`에 저장되고, 파일 경로가 사용자에게 제공됨 |
| **검증 방법** | 스키마 검증 (파일 존재 확인, 용량 > 0) |
| **실패 시 처리** | 저장 실패 시 자동 재시도 1회 → 재실패 시 에스컬레이션 (사용자에게 경로/권한 확인 요청) |

#### 세션 복원

| 항목 | 내용 |
|------|------|
| **트리거** | 세션 시작 시 항상 실행 |
| **로직** | /output 디렉터리에서 파일 존재 여부 확인 → step4_draft.md 있으면 중간 확인 2부터, step3_strategy.json 있으면 Phase 4부터, step2_research.json 있으면 중간 확인 1부터, step1_context.json 있으면 Phase 2부터, 없으면 처음부터 |
| **사용자 확인** | 재개 시 "이전 작업이 발견되었습니다. [단계]부터 이어하겠습니다. 처음부터 다시 시작할까요?" 메시지 출력 |

1. 역할 및 목적 정의 (오케스트레이터, 개인 생산성 도구)

3. 전체 워크플로우 순서 및 중간 확인 트리거 조건
4. 서브에이전트 호출 방법 및 데이터 전달 규약
5. 인터뷰 선정 로직 (미니 스캔 → 부족한 정보만 질문)
6. **인터뷰 결과 확인 메시지 포맷 (기본값 명시 포함)**
7. 중간 확인 메시지 포맷 (발견 가설 3개 + 근거, 충돌 명시)
8. 실패 처리 및 에스컬레이션 기준 (사용자에게 추가 정보 직접 요청)
9. 최종 산출물 저장 경로 및 네이밍 규칙

> 채택 근거: Phase마다 필요한 지식(웹검색·결과평가 / 전략프레임워크 / 문서작성)이 서로 달라 하나의 컨텍스트에 모두 로드하면 비효율적. 각 에이전트가 해당 단계에서만 활성화되어 컨텍스트 윈도우를 최적화.

| 에이전트 | 역할 | 트리거 조건 | 입력 | 출력 |
|---------|------|------------|------|------|
| **CLAUDE.md** (메인) | 전체 오케스트레이션, 인터뷰 수행, 중간 확인 관리, 세션 복원 | 항상 활성 | 사용자 키워드 | 워크플로우 진행, 최종 파일 경로 제공 |
| **researcher** | 웹 검색 3개 트랙 **병렬** 실행, 결과 평가·구조화, 충돌 감지 | Phase 2 시작 시 | `step1_context.json` | `step2_research.json` |
| **strategist** | SWOT·포지셔닝·전략 옵션 도출, Two-pass 검증 | 중간 확인 1 승인 후 | `step2_research.json` | `step3_strategy.json` |
| **writer** | 산출물 유형별 문서 초안 생성, Two-pass 품질 검증, FEEDBACK_LOG 반영 | Phase 4 시작 시 | `step1_context.json` + `step2_research.json` + `step3_strategy.json` | `step4_draft.md` |

**서브에이전트 간 통신 원칙**: 직접 호출 금지 → 메인(CLAUDE.md)을 통해 파일 경로 전달 방식으로만 조율

### 3.5 서브에이전트 상세

#### researcher (AGENT.md 핵심 내용 목록)

- 역할: 시장·경쟁사·외부환경 3개 트랙 **병렬** 웹 검색 및 결과 평가
- 트리거: 메인 에이전트가 `step1_context.json` 경로를 전달하며 호출
- 입력: `step1_context.json` (키워드, 산출물 유형, 분석 범위, freshness_months)
- 출력: `step2_research.json` (트랙별 핵심 데이터, 출처 URL, 신뢰도 메모, 충돌 플래그)
- 참조 스킬: `web-search`, `json-builder`
- **병렬 실행**: 3개 트랙을 순차가 아닌 동시에 실행 (AGENT.md에 명시)
- **쿼리 설계 원칙**: 트랙당 5회 제한 내에서 정보량·커버리지 기준으로 쿼리 설계
- 자기 검증 기준: 출처 3건 이상, freshness_months 이내, 3개 트랙 모두 데이터 존재
- **충돌 감지**: 사용자 의도(step1_context의 direction)와 수집 데이터가 충돌하면 research.json의 `conflict_flag`로 표시

#### strategist (AGENT.md 핵심 내용 목록)

- 역할: 리서치 데이터 기반 SWOT, 포지셔닝, 전략 옵션 3개 도출
- 트리거: 메인 에이전트가 중간 확인 1 승인 후 `step2_research.json` 경로 전달
- 입력: `step2_research.json`
- 출력: `step3_strategy.json` (SWOT 4분면, 포지셔닝 분석, 전략 옵션 3개 + 근거)
- 참조 스킬: `json-builder`
- **Two-pass 검증**: 1차 생성 완료 후 동일 에이전트가 비판자 역할로 전환하여 재검토
- **전략 다양성 제약**: 3개 옵션은 리스크·투자 규모·진입 타이밍 기준으로 서로 구별되어야 함. 동질적 옵션 금지.
- 자기 검증 기준: SWOT 각 항목 2개 이상, 전략 옵션마다 데이터 기반 근거 명시, 3개 옵션 간 명확한 차별성

#### writer (AGENT.md 핵심 내용 목록)

- 역할: 산출물 유형에 맞는 문서 초안 생성 및 품질 Two-pass 검증
- 트리거: 메인 에이전트가 Phase 4 진입 시 context + research + strategy 파일 경로 전달
- 입력: `step1_context.json` + **`step2_research.json`** (원본 수치·출처 직접 인용) + `step3_strategy.json`
- 출력: `step4_draft.md` (하단에 `<!-- FEEDBACK_LOG -->` 섹션 포함)
- 참조 스킬: `doc-formatter`
- 참조 문서: `/docs/document_templates.md` (유형별 섹션 구조)
- **FEEDBACK_LOG 처리**: 재실행 시 step4_draft.md의 `<!-- FEEDBACK_LOG -->` 섹션을 먼저 읽어 모든 피드백 이력 파악 후 반영
- **Two-pass 검증**: 초안 생성 후 비판자 역할로 전환하여 섹션 완성도·톤 일관성·독자 적합성 재검토
- 자기 검증 기준: 필수 섹션 존재, 각 섹션 200자 이상, 독자 적합 톤, FEEDBACK_LOG의 모든 지시 반영 확인

### 3.6 데이터 전달 패턴

```
메인 에이전트
  [시작 시] /output 파일 존재 체크 → 재개 시점 결정

  → researcher 호출: step1_context.json 경로 전달 (파일 기반)
  ← researcher 완료: step2_research.json 경로 반환

  → 중간 확인 1: step2_research.json 기반 발견 가설 3개 + 근거 인라인 제시
     (conflict_flag 있으면 충돌 내용 + 방향 선택 요청)
  ← 사용자 승인/피드백: 채팅 인터페이스

  → strategist 호출: step2_research.json 경로 전달 (파일 기반)
  ← strategist 완료: step3_strategy.json 경로 반환

  → writer 호출: step1_context.json + step2_research.json + step3_strategy.json 경로 전달 (파일 기반)
  ← writer 완료: step4_draft.md 경로 반환

  → 중간 확인 2: step4_draft.md 전체 내용 제시 (인라인 렌더링)
  ← 사용자 피드백: 채팅 인터페이스

  [피드백 있으면] step4_draft.md의 FEEDBACK_LOG에 이력 추가
  → writer 재호출: step4_draft.md(FEEDBACK_LOG 포함) 경로 전달
  ← writer 완료: 수정된 step4_draft.md

  → doc-formatter 스킬: step4_draft.md → final_<type>_<date>.md 저장
```

### 3.7 주요 산출물 파일 형식

| 파일 | 형식 | 용도 |
|------|------|------|
| `step1_context.json` | JSON (필수 필드: `keyword`, `output_type`, `audience`, `constraints`, **`freshness_months`**) | 인터뷰 결과 저장 |
| `step2_research.json` | JSON (필수 필드: `market`, `competitors`, `external`, `sources`, **`conflict_flag`**) | 리서치 결과 저장 |
| `step3_strategy.json` | JSON (필수 필드: `swot`, `positioning`, `options[3]`) | 전략 분석 저장 |
| `step4_draft.md` | Markdown (하단에 `<!-- FEEDBACK_LOG -->` 섹션 포함) | 문서 초안 + 피드백 이력 |
| `final_<type>_<date>.md` | Markdown 전용 (docx는 v2) | 최종 사용자 산출물 |

## 4. 확정된 결정사항

인터뷰를 통해 기존 미결 결정사항이 모두 확정되었습니다.

| 항목 | 결정 | 근거 |
|------|------|------|
| 웹 검색 도구 | **Claude 내장 WebSearch** | 추가 설치 불필요, Tavily는 v2 확장 |
| 최종 문서 포맷 | **Markdown 전용** | 개인 도구, docx 변환은 v2 |
| 병렬 검색 구현 | **researcher 내 병렬 Task 동시 실행** | 속도 필수 요건 |
| 최신성 기준 | **인터뷰 시 freshness_months 동적 설정** | 분야별 변화 속도 상이 |
| 자기 검증 방식 | **Two-pass (생성 + 비판자 역할 전환)** | 단순 자기 검증보다 신뢰도 높음 |
| Writer 입력 | **context + research + strategy 3종** | 원본 수치·출처 인용 필요 |
| 피드백 루프 | **FEEDBACK_LOG 누적 방식** | 이전 피드백 회귀 방지 |
| 세션 복원 | **/output 파일 존재 체크 자동 탐지** | 크래시·중단 시 재작업 방지 |
| v1 주력 문서 | **기획서** | 가장 자주 사용하는 유형 집중 |
| 에스컬레이션 | **사용자에게 추가 정보 직접 요청** | 가진 정보로 추진보다 투명성 우선 |
